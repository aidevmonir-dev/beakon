## Sheet: `Dimension Ref expl.`

_160 rows x 1 cols_

- A few notes on the design
- 1. Why Dimension_Type_Code matters
- This should be the stable system code.
- Example:
- PORT
- CUST
- INST
- CCY
- These codes should be what Beakon uses in rules and metadata, not the long names.
- 2. Why Applies_To matters
- This tells the system where the dimension is relevant.
- Example:
- INST applies to Investment
- FAM applies to Personal Expense;Reporting
- TRF applies to Transfers;Mapping;Controls
- This will later help Beakon decide:
- which dimensions to prompt for
- which dimensions are relevant for which account classes
- which validation rules to apply
- 3. Why Mandatory_Flag is important
- This is strategic.
- For example:
- Currency should clearly be mandatory
- Instrument should be mandatory for investment postings
- Family Member should usually not be globally mandatory, even if it is useful for school fees
- So this flag is not about one specific posting yet.
- It is about whether the dimension is fundamentally required in the model.
- Later, if needed, account-level logic in 02_CoA_Master can override this more specifically.
- 4. Why Hierarchy_Allowed matters
- Some dimensions need parent-child logic.
- Examples:
- Portfolio
- PORT_MAIN
- PORT_EQ
- PORT_ALT
- PORT_PRIV
- Related Party
- RP_FAMILY_OFFICE
- RP_HOLDCO_001
- RP_TRUST_001
- Currency
- Usually no hierarchy
- That is why Hierarchy_Allowed should sit at type level.
- 5. Why Master_Data_Owner matters
- This is operationally important from the beginning.
- Examples:
- System for Currency
- Finance for Portfolio / Custodian / Transfer Type
- Investment Ops for Instrument / Tax Lot / Commitment
- Client Admin for Family Member
- This becomes useful later when Beakon needs approval workflows or restricted edit rights.
- How this links to the other tabs
- Link to 02_CoA_Master
- In 02_CoA_Master, a row may say:
- Dimension_Set = Instrument;Custodian;Portfolio;Currency;Tax Lot
- Those dimension names must exist here as dimension types:
- INST
- CUST
- PORT
- CCY
- TLOT
- Link to future 05_Dimension_Values
- This tab defines the type.
- The next tab will define the values.
- Example:
- Here in 04_Dimension_Types
- CUST | Custodian
- Later in 05_Dimension_Values
- CUST | CUST_UBS_ZRH | UBS Zurich
- CUST | CUST_PICTET_GVA | Pictet Geneva
- That is the clean split.
- My recommendations before you confirm this tab
- I would suggest you review 3 things in particular:
- 1. Whether Report Category should really be a dimension
- I included it because it existed in the earlier draft logic, but it is more of a reporting metadata classification than a true transactional dimension.
- So this one is optional.
- 2. Whether Asset Class should be a true dimension or derived from instrument master
- Long-term, I would normally derive asset class from the instrument master, not ask users to post both separately.
- But for early flexibility, it can stay as a dimension type.
- 3. Whether Client Bank Account should stay distinct from Custodian
- I think yes.
- Reason:
- custody accounts and bank accounts behave differently operationally
- one is for portfolio custody
- one is for payment and cash movement workflows
- So I would keep BANK separate from CUST.

## Sheet: `Dimension values expl.`

_113 rows x 1 cols_

- How this now works with the rest of the model
- 04_Dimension_Types
- Defines the category:
- PORT

- INST

- 05_Dimension_Values
- Defines the members inside each category:
- PORT_MAIN
- CUST_UBS_ZRH
- INS_NESN

- 02_CoA_Master
- References the dimension types in the Dimension_Set field.
- Example:
- 130100 Listed equities
- Dimension_Set = Instrument;Custodian;Portfolio;Currency;Tax Lot
- Later, those labels should ideally become the type codes:
- INST;CUST;PORT;CCY;TLOT
- That is one clean-up I recommend for the next pass.
- My main recommendations on this tab
- 1. Keep codes stable
- Names can change. Codes should not.
- For example:
- CUST_UBS_ZRH should remain stable even if you later rename the display name from “UBS Zurich” to something longer.
- 2. Use parent-child only where useful
- This is especially helpful for:
- portfolios
- related parties
- maybe properties
- maybe reporting categories
- It is not very useful for:
- currency
- transfer type
- tax lot
- 3. Treat some of these as starter values only
- This draft intentionally includes:
- example instruments
- example counterparties
- example policies
- Those are fine to get the structure right, but later Beakon may want separate master data tables for:
- instrument master
- counterparty master
- policy master
- 4. RCAT is optional
- I included RCAT because it was in the earlier logic, but it is more of a reporting classification layer than a transactional dimension.
- It is fine to keep for now, but I would treat it as optional.
- Important refinement I suggest next
- In the next cleanup pass, I would align 02_CoA_Master so that Dimension_Set uses the type codes instead of the long names.
- Example:
- instead of:
- Instrument;Custodian;Portfolio;Currency;Tax Lot
- use:
- INST;CUST;PORT;CCY;TLOT
- That will make the workbook much more system-ready.
- Where you now stand
- You now have the right structural model:
- 01_CoA_Definition
- 02_CoA_Master
- 03_CoA_Mapping
- 04_Dimension_Types
- 05_Dimension_Values
- That is a much stronger starting point for Beakon than a flat CoA-only approach, because it gives you:
- scalable accounting structure
- proper dimension logic
- future mapping support
- room for multicurrency and sub-ledger depth
- less redesign later
